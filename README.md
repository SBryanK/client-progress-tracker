# Client Progress Tracker

A weekly client progress tracker built with **Next.js 15**, **Prisma**, and **SQLite** — a web-app replacement for the Word-based client tracker. Works on desktop, tablet, and mobile browsers.

> Generated by the `project-creator` skill following the full 9-phase pipeline
> (Brainstorm → Clarify → Architect → Select Stack → UI/UX → Build → Test → Deploy → Report).

---

## ✨ Features

- **Public landing page** — anyone can read the site without signing up. The landing page
  shows stats, status distribution, and the latest weekly updates. No barrier for managers
  or colleagues who just want to glance at progress.
- **Hidden sign-in** — a discrete "Sign in" button in the top-right corner opens a modal
  dialog. Only pre-authorized emails (see `.env`) can sign in. There is no public sign-up.
- **Dashboard** (owner-only) — stats, status distribution, health-score roll-up, latest weekly updates.
- **Clients** — full CRUD for owners, public read-only for everyone else. Search + filter
  by status/priority/text, sort by health.
- **Client detail** — activity/issue/progress log, weekly-update history, public read-only;
  edit/delete buttons only render for the signed-in owner.
- **Weekly updates timeline** — public read-only, grouped by week across all clients.
- **Share links** (owner-only, at `/dashboard/share`) — generate unguessable read-only URLs
  scoped by client, date range, or status. Revocable, with optional expiry.
- **Reports** — Word / PDF / Excel / Markdown export for 8 scopes
  (weekly all, monthly all, client, status, active only, high priority, blockers, manager summary).
- **Import** — drop your existing `.docx` or `.xlsx` tracker; the parser extracts clients,
  weekly bullets, and the "List of Works" section automatically.
- **Multi-owner access** — multiple emails can be listed as OWNER in `.env`; every OWNER
  can read/write everything. All role checks enforced on the backend (API routes).
- **Transparent health score** — 0–100 with a per-factor breakdown you can see on the client page.
- **Dark mode** — FOUC-free, respects `prefers-color-scheme`.
- **Accessibility** — WCAG AA, labeled inputs, focus-visible rings, `prefers-reduced-motion`.

## 🚀 Quick start

```bash
cd /Users/sbryankusno/Documents/Akamai-Migration/bryan/apps/client-progress-tracker

# 1. Install
npm install            # or: pnpm install

# 2. Configure
cp .env.example .env
# Edit .env — set OWNER_EMAILS (comma-separated), OWNER_PASSWORD, AUTH_SECRET.
# Generate AUTH_SECRET: openssl rand -base64 32

# 3. Database
npm run db:push        # creates prisma/dev.db from schema
npm run db:seed        # creates every OWNER_EMAILS account

# 4. (Optional) Import your existing Word tracker
npm run import:docx "/Users/sbryankusno/Documents/Akamai-Migration/bryan/Bryan's+Client+Involvement+Progress+(Intern+Edition).docx"

# 5. (Optional) Reassign existing clients to the new primary owner
# Use this if you changed OWNER_EMAILS after already seeding & importing —
# it moves every Client that belongs to a demoted user over to the new
# primary OWNER_EMAILS[0]. Safe to re-run.
npm run db:reassign-owner

# 6. Run
npm run dev            # http://localhost:3000
```

Visit `http://localhost:3000` — the landing page is public. Click **Sign in** (top-right)
and use any email from `OWNER_EMAILS` + the shared `OWNER_PASSWORD`.

## 🔑 Role model (enforced on the backend)

The site is **public-by-default**: every anonymous visitor can read. The only gated
surfaces are the write-capable ones (dashboard, import, reports, share-link manager,
client create/edit/delete).

`OWNER_EMAILS` in `.env` is a comma-separated list of emails that can sign in and edit.
Anyone not on that list is an anonymous reader — no account needed.

| Capability                          | OWNER | Anonymous |
| ----------------------------------- | :---: | :-------: |
| View landing / clients / weekly     |   ✅   |    ✅     |
| View per-client share link          |   ✅   |    ✅     |
| Add / edit / delete clients         |   ✅   |    ❌     |
| Add weekly updates & activities     |   ✅   |    ❌     |
| Import Word / Excel                 |   ✅   |    ❌     |
| Export reports                      |   ✅   |    ❌     |
| Create / revoke share links         |   ✅   |    ❌     |

Role is baked into the NextAuth JWT; every mutating API route calls
`requireOwner()` before doing work. Middleware blocks anonymous GETs on all
owner-only UI routes and bounces them to `/?signin=1&next=...` so the
embedded sign-in dialog auto-opens.

## 🧠 Client Health Score (transparent & explainable)

| Band       | Score | Meaning              |
| ---------- | :---: | -------------------- |
| Healthy    | 80–100 | Green light          |
| Needs attention | 60–79 | Watch it           |
| At risk    | 40–59 | Unblock soon         |
| Blocked    | 0–39  | Immediate triage     |

Factors (weights shown on the client page):

| Factor          | Max |
| --------------- | :-: |
| Update recency  | 30  |
| Status          | 25  |
| Blockers clear  | 15  |
| Next action set | 10  |
| Momentum (30d)  | 10  |
| Priority        | 10  |

See `src/lib/health.ts` for the exact scoring.

## 📥 Import — how the .docx parser works

The parser (`src/lib/import-docx.ts`) mirrors the real template:

1. Everything before the `List of Works` heading is the **weekly-updates** half.
   - Date-range paragraphs (e.g. `April 6 – April 10`) start a new week.
   - Non-bullet paragraphs become client sub-headings.
   - `•` / `-` lines become that client+week's bullets.
2. Everything after `List of Works` is the **activities** half.
   - Client heading, then `Activities` / `Issues/Support` / `Progress` sub-sections.
3. Status is inferred from inline parentheticals (`(Ongoing)`, `(NEW LEAD)`,
   `(Terminated)`, `(Low Priority)`, `(Stuck)`, `(Shadowing)`, …).

## 📤 Export formats

- **Word (`.docx`)** — headings, bullets, tables, client grouping. Uses `docx` npm package.
- **PDF** — via `jspdf` + `jspdf-autotable`. Includes summary table + per-client sections.
- **Excel (`.xlsx`)** — two sheets: full Report and Summary. Uses `xlsx`.
- **Markdown** — plain text, ideal for pasting into Slack / Notion / iWiki.

All drivers live in `src/lib/reports/` and share a single `ReportData` type.

## ☁️ Deployment

### Option 1 — Vercel (recommended for demo)

```bash
npm i -g vercel
vercel
```

In the Vercel dashboard, set:
- `DATABASE_URL` — swap to Neon / Turso / Supabase Postgres (SQLite won't work on serverless).
- `AUTH_SECRET`, `AUTH_TRUST_HOST=true`, `NEXT_PUBLIC_APP_URL=https://your-app.vercel.app`
- `OWNER_EMAIL`, `OWNER_PASSWORD`, `VIEWER_EMAILS`, `VIEWER_DEFAULT_PASSWORD`

Then update `prisma/schema.prisma` `provider` to `postgresql`, and run `prisma migrate deploy`.

### Option 2 — Railway / Fly.io

`package.json`'s `build` already runs `prisma migrate deploy`. Expose port `3000`, set the
same env vars, attach a Postgres add-on, and you're done.

### Option 3 — Keep SQLite on a VPS (cheapest)

Works out of the box with `node server.js` (`npm run start`) behind Nginx/Caddy. The
SQLite file lives at `prisma/dev.db`; back it up on a schedule.
